## What this is

**Stage 6c addresses three surfaces of throws PR card visual polish.** The build deliverables span CSS, asset moves, and one new asset that Oak generates. After 6c, the throws PR card reads as a bold visual moment — silhouette dominates, text reads as a cohesive block, no orphan whitespace, Saltire flag fills the negative space as atmospheric background.

The three surfaces:

1. **Silhouette size relaxation and text cohesion** — the silhouette is currently capped at `height: 74%` and `max-width: 68%`, which leaves it small for both wide images (hammer; width-bound rendering with vertical whitespace) and tall images (sheaf, WOB; height-bound rendering with horizontal whitespace, and a thin vertical-standards artwork that reads as empty). Plus the text layout creates a "valley" of whitespace between the mark and the meta. Both relax.

2. **Scottish Saltire flag background** — a new atmospheric image element behind the silhouette, filling the otherwise-empty card area. Subtle (low opacity), consistent across all five implements, generated by Oak via the same GPT-image-gen workflow used for the silhouettes.

3. **WOB silhouette refresh** — Oak regenerated the WOB adaptive variant 2026-05-27 evening with the weight clearly above the bar (clear air between them). The refreshed asset is staged in `Images for Cards/silhouette-weight-over-bar-adaptive.png`; ccode moves it to `images/silhouettes/`. The able-bodied WOB may need the same refresh; Oak decides whether to regenerate it inside 6c or land it as a small follow-on.

## Stage 6c scope

### Part A — CSS / layout (ccode-buildable, no new assets)

1. **Relax `.throw-silhouette` size constraints** in `styles.css` so the silhouette dominates the card visually. Suggested starting values:
   ```css
   .throw-silhouette {
     position: absolute;
     right: 0;
     bottom: 0;
     height: 95%;          /* was 74% */
     width: auto;
     max-width: 92%;       /* was 68% */
     object-fit: contain;
     object-position: bottom right;
     z-index: 0;
     pointer-events: none;
   }
   ```
   These are starting points, not locked values — ccode is free to nudge to whatever produces the best visual across all 10 silhouettes (5 implement pairs). Verify against each implement; tall narrow ones (sheaf, WOB) and wide ones (hammer) will both need to read well.

2. **Tighten text grouping** so the mark + previous-mark + date + games title + location read as one cohesive block, not separated by orphan whitespace. The current flex/grid layout pushes the metadata to the bottom of the card with a void in the middle. Move the metadata block up so it sits directly under the mark with normal line spacing, leaving the bottom of the card for the wordmark and any horizontal divider.

   The intent: the text is a tight stack in the upper-left, the silhouette is the visual anchor in the lower-right (and now spills further across the card), the wordmark sits at the bottom edge as it does today.

3. **Z-index discipline preserved.** Text must remain above silhouette (current `.celebration-card--throw > :not(.throw-silhouette) { position: relative; z-index: 1; }` rule stays). With the silhouette larger, text-over-silhouette overlap becomes more likely — confirm the dark text reads clearly over the black silhouette areas where they overlap (the silhouette is high-contrast black-on-white; text overlap on the silhouette areas is the concern). If a subtle text-shadow or background-color stripe behind text is needed for legibility, add it.

### Part B — Scottish Saltire flag background asset

4. **Stage a new background asset** at `images/decorations/saltire.png` (new directory; net-new asset). **Cowork-generated programmatically** from the public-domain Saltire spec (Scottish national flag — white X on blue background, Pantone 300 equivalent `#0065BD`, stripe width 1/5 of the flag's shorter dimension, diagonal cross extending fully into all four corners). 1500×1800 portrait so it fits a 4:5 card under `background-size: cover` with minimal cropping. One image used across all implements; same image, same position on every throws PR card.

   *(Cowork's asset, not Oak's. Generated 2026-05-27 late session via PIL polygon-fill to dodge license risk on stylized stock-rendered Saltires; the flag design itself is public-domain but specific waving/photoreal renderings often aren't. The wave motion comes from CSS, not from the asset — the asset is a flat geometric flag at rest.)*

5. **Add CSS for the flag background** as a new layer behind the silhouette but above the card background. The flag waves for ~5 seconds on card render, then freezes:
   ```css
   .celebration-card--throw::before {
     content: '';
     position: absolute;
     inset: 0;
     background: url('../images/decorations/saltire.png') no-repeat;
     background-size: cover;        /* or contain, depending on the asset aspect */
     background-position: center;
     opacity: 0.22;                 /* atmospheric, not foreground */
     z-index: 0;                    /* same layer as silhouette base, silhouette wins via DOM order */
     pointer-events: none;
     transform-origin: center;
     animation: saltire-wave 5s ease-out forwards;
     animation-iteration-count: 1;
   }
   @keyframes saltire-wave {
     0%   { transform: skewX(-3deg) skewY(0deg); }
     20%  { transform: skewX(3deg)  skewY(-1deg); }
     40%  { transform: skewX(-2deg) skewY(1deg); }
     60%  { transform: skewX(2deg)  skewY(-0.5deg); }
     80%  { transform: skewX(-1deg) skewY(0.5deg); }
     100% { transform: skewX(0deg)  skewY(0deg); }
   }
   /* Respect reduced-motion preference — settle straight to the frozen state. */
   @media (prefers-reduced-motion: reduce) {
     .celebration-card--throw::before {
       animation: none;
     }
   }
   .throw-silhouette {
     z-index: 1;                    /* bump above the flag pseudo */
   }
   .celebration-card--throw > :not(.throw-silhouette) {
     z-index: 2;                    /* text stays on top of both */
   }
   ```
   The animation runs once each time the celebration card mounts — both at save-time and on Past Sessions replay. `animation-fill-mode: forwards` (via the `forwards` shorthand) holds the final frame after the 5 seconds elapse. Opacity, animation curve, keyframe values, position, and size are starting points — Oak will iterate visually once he sees the flag in the card. The acceptance bar is: silhouette stays the hero, flag waves naturally for ~5s then settles, soft-grey card base still reads, text is fully legible over any combination at every animation frame.

## Tech notes

- Vanilla HTML/CSS/JS, no build step. No new dependencies. The Saltire wave is pure CSS keyframes — no JS animation library, no requestAnimationFrame loop.
- The flag asset Oak generates is **one static PNG of the frozen final state** of the wave (the flag at rest after curling). CSS keyframes then apply transform-skew over 5 seconds to produce the wave motion, settling on the unskewed final frame. Oak does NOT need to produce an animated APNG/GIF/Lottie — single PNG is the input.
- `animation-iteration-count: 1` + `animation-fill-mode: forwards` (folded into the `forwards` shorthand) is the load-bearing pair — without `forwards`, the element snaps back to the start state after the animation ends.
- The animation fires on every card mount. This means a user who replays a celebration from Past Sessions will see the wave run each time — that's intentional.
- `prefers-reduced-motion: reduce` accessibility guard is required, not optional. Vestibular-disorder users will get sick if a screen-filling background skews unprompted.
- No schema change, no migration, no storage touched, no test changes expected.
- The new `images/decorations/` directory is net-new; ensure it exists before the asset stage commit.
- Z-index layering is the load-bearing piece of the CSS work — get this right and the rest follows.

## Risk note

Project risk is **Moderate.** Higher than 6a/6b for two reasons:

1. **Design iteration may be needed.** The CSS values (silhouette dimensions, flag opacity/position, wave animation curve/keyframes) are starting points; the spec doesn't lock them. Oak's eyeball is the final arbiter. ccode should plan on Oak reviewing the build on his phone and possibly requesting adjustments — the build session may include 1-2 design iterations.

2. **Z-index and text-legibility regressions.** Making the silhouette bigger increases the area where text might overlap dark silhouette regions. The build needs to confirm legibility on each implement, not just one. The Saltire background at low opacity should not affect text legibility but is one more layer to check. Mid-animation legibility also needs checking — at any keyframe the text must still read clearly.

3. **Animation-induced motion-sickness regression.** The `prefers-reduced-motion` guard is load-bearing. If it's omitted or written incorrectly, vestibular-disorder users get a screen-filling skew animation they can't escape. ccode must verify the media query works (toggle Reduce Motion in macOS System Settings → Accessibility → Display, reload, confirm the flag is frozen with no animation).

The codex review's focus:

- The silhouette size relaxation actually produced a more dominant figure across all 10 implements (no regressions where the change made an implement look worse).
- Text remains legible against silhouette + flag + card background overlap on every implement — at every frame of the wave animation, not only the final frozen state.
- The Saltire image is at the correct path (`images/decorations/saltire.png`) and renders.
- The Saltire wave animation runs once for ~5 seconds and freezes on the final frame (verify the `forwards` fill-mode and `iteration-count: 1` are both present).
- The `prefers-reduced-motion: reduce` media query suppresses the animation correctly.
- Both refreshed WOB silhouettes (adaptive and able-bodied) are at the correct shipped paths and show the weight clearly above the bar.
- No regressions on lift PR cards, Goal cards, or Awesome Day capstones (scope isolation preserved).
- No test changes (this is presentation-only) and `npm run lint` clean.
- The existing `--no-silhouette` defensive fallback still works if the silhouette or flag image fails to load.

## Open items
