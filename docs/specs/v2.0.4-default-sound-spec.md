# Stone & Standard v2.0.4 — default sound ON + profile preference

Small patch flipping celebration audio to default-ON, plus adding
a sound preference control to the profile modal (accessible via the
Edit Profile button shipped in v2.0.1). The celebration overlay's
existing instant-toggle stays — users now have two interchangeable
access points to the same preference.

## Known risks carried

⚠️ **`stone-and-standard-sound` standalone storage key** — kept
separate from the `data.profile` blob per Oak's explicit choice
2026-05-29 to avoid a schema migration. Trigger condition: if the
standalone key read/write fails, syncs poorly with profile data,
breaks backup-export coverage, or otherwise causes friction, the
mitigation is a schema bump (v2 → v3) that moves the preference
into `profile.soundOn` with a full migration pass. This block
travels with every spec from v2.0.4 forward until the risk either
triggers (do the migration) or is explicitly retired after a
documented stability period. See [[reference_carried-risks-convention]]
for the pattern, [[project_highland-games-tracker]] for the
specific risk record.

## Risk overlay

**Normal** — UI + one-character logic change in `isSoundOn`. No
schema bump (per the carried risk above). Codex review on, one
pass.

## Design decisions locked

- **Default sound: ON.** New installs and any user whose
  localStorage doesn't contain an explicit `'off'` get sound on.
  Users who actively toggled sound off keep their choice.
- **Save behavior on the profile checkbox: immediately on toggle.**
  The checkbox writes to `localStorage` the moment the user clicks
  it, via `setSoundOn(checked)`. NOT coupled to the form's Save
  button. The form's Save button still commits the rest of the
  profile fields (name, gender, weight schedule, class, tier).
- **Storage location: standalone key** (NOT `profile.soundOn`).
  See the Known risks carried block above for the trigger
  condition that would change this.
- **Sync between the two access points** (celebration overlay
  toggle and profile modal checkbox): automatic via the shared
  `isSoundOn` / `setSoundOn` helpers. No additional wiring needed.

## What changes

### `session.js` — flip `isSoundOn`'s default

Single-character logic change:

```js
function isSoundOn() {
  try {
-   return localStorage.getItem(SOUND_PREF_KEY) === 'on';
+   return localStorage.getItem(SOUND_PREF_KEY) !== 'off';
  } catch {
    return false;
  }
}
```

Truth table:

| Stored value | Pre-v2.0.4 returns | v2.0.4 returns |
|---|---|---|
| absent (key not set) | `false` | **`true`** |
| `'on'` | `true` | `true` |
| `'off'` | `false` | `false` |
| anything else | `false` | `true` |

The "anything else returns true" case is intentional defensive
behavior — corrupted localStorage values default to sound-on rather
than sound-off, matching the new opt-out posture.

The private-mode catch (storage disabled) still returns `false`
— if storage is genuinely broken the preference can't be honored
anyway, defaulting to silent in that edge case is safer than
forcing audio.

### `index.html` — add the profile modal sound checkbox

Inside the existing `#profile-modal` markup, add a new row at the
bottom of the form (just above the Save button). Suggested
structure:

```html
<div class="profile-sound-row">
  <label class="profile-sound-label">
    <input
      type="checkbox"
      id="profile-sound-toggle"
      aria-describedby="profile-sound-help"
    >
    <span>Play celebration sounds</span>
  </label>
  <p id="profile-sound-help" class="profile-sound-help">
    Sound stays off if you turn it off here. The overlay on a
    celebration card has the same toggle.
  </p>
</div>
```

Placement: directly above the existing Save button, separated by
the same visual rhythm as the other profile field rows. The help
text is short and non-blocking — appears underneath the checkbox
so screen readers get the context via `aria-describedby` and
sighted users get the parenthetical-style note.

### `app.js` — wire pre-population AND save-immediately-on-toggle

Two additions to `openProfileModal` / `setupProfileModal`:

**Pre-population in `openProfileModal`**, alongside the existing
field pre-pop block:

```js
const soundCheckbox = document.getElementById('profile-sound-toggle');
if (soundCheckbox) {
  soundCheckbox.checked = isSoundOn();
}
```

**Change listener in `setupProfileModal`**, attached once at init
(same pattern as the existing class / gender listeners):

```js
const soundCheckbox = document.getElementById('profile-sound-toggle');
if (soundCheckbox) {
  soundCheckbox.addEventListener('change', () => {
    setSoundOn(soundCheckbox.checked);
  });
}
```

The save-immediately-on-toggle pattern means **the existing form
submit handler doesn't touch the sound preference at all**. The
checkbox saves on toggle, full stop. The Save button keeps doing
exactly what it did before for the profile-data fields.

### `styles.css` — minimal layout for the new row

Add a small rule for `.profile-sound-row` matching the existing
profile field row's vertical rhythm. Help text should be muted and
smaller than the label. Specifics:

```css
.profile-sound-row {
  margin-top: 1rem;
  padding-top: 1rem;
  border-top: 1px solid var(--border);
}
.profile-sound-label {
  display: flex;
  align-items: center;
  gap: 0.5rem;
  font-weight: 500;
}
.profile-sound-help {
  margin: 0.35rem 0 0 1.5rem;
  font-size: 0.78rem;
  color: var(--muted);
}
```

The top border + padding separates the preference row from the
profile-data fields above it, visually reinforcing that it's a
preference, not a profile field.

## Tests in `tests.js`

Three new tests for the `isSoundOn` logic flip:

```js
test('isSoundOn: returns true when key is absent (v2.0.4 default-ON)', () => {
  localStorage.removeItem(SOUND_PREF_KEY);
  assertTrue(isSoundOn());
});

test('isSoundOn: returns true when key is "on"', () => {
  localStorage.setItem(SOUND_PREF_KEY, 'on');
  assertTrue(isSoundOn());
});

test('isSoundOn: returns false when key is explicitly "off"', () => {
  localStorage.setItem(SOUND_PREF_KEY, 'off');
  assertEqual(isSoundOn(), false);
});
```

The profile-modal-DOM tests are skipped per the v2.0.1 pattern
(harness doesn't render the modal markup; the
pre-population / change-listener behavior is covered by manual
smoke testing).

Suite count: 386 → 389.

**Manual smoke test additions** (added to the existing checklist):

- Fresh install (clear localStorage, reload). Hit any PR. Verify
  audio plays without the user having toggled anything (default-ON
  confirmed).
- Open profile modal via the Edit Profile button. Confirm the
  sound checkbox shows checked (matching default-ON).
- Toggle the sound checkbox OFF in the profile modal. Without
  hitting Save, close the modal. Hit a PR. Verify silence (save-
  immediately-on-toggle confirmed).
- Toggle sound ON via the celebration overlay's existing button on
  a celebration card. Open the profile modal. Verify the checkbox
  reflects the new state (sync between access points confirmed).
