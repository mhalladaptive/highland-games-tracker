# Stone & Standard v2.0.1 — patch release

Small patch release surfaced from first-hour user testing of the
shipped v2.0 deployment. Three concerns bundled, no data-layer
touch, all UI / CSS only.

## What's in the bundle

1. **Edit Profile affordance** on the Set PRs & Goals page, with a
   small modal refactor so re-opening pre-populates from existing
   profile data.
2. **Throws PR card text overlays nudge down** a fingernail (~1–2%)
   so the plaque text and scroll metadata sit centered within their
   illustrated regions on the template. Plaque and scroll *positions*
   stay put; only the *text* inside each moves.
3. **Stone & Standard wordmark font shrinks** from `0.7rem` to
   `0.525rem` (–25%) on the ornate throws PR card.

## Risk overlay

**Normal** — no data deletion, no auth, no money. UI + CSS only.
Codex review on per Level 1 baseline (one pass).

## Concern 1 — Edit Profile affordance

### What's wrong today

The profile modal opens only at first launch when
`setupCompletedAt` is unset (`app.js:610`). A user who dismisses the
modal without finishing — even by accidentally clicking off — is
stuck: there's no path to re-open the modal anywhere in the app
surface. Oak hit this on the production GitHub Pages URL within
minutes of launch.

### What changes

A small **Edit Profile** button at the top of the Set PRs & Goals
page. Gear icon (`⚙` Unicode, no asset dependency) plus an explicit
"Edit profile" text label so the affordance is self-describing. Top
of `<main>`, right-aligned, above the existing intro paragraph.
Visible from any state — first-launch dismissed, profile fully
completed, anything in between.

Clicking the button opens the existing modal, pre-populated with
whatever's currently in `data.profile`. Save behaves as today —
writes back to `data.profile` via `saveData`.

### Modal refactor (enables re-edit-safety)

The current `openProfileModal()` at `app.js:577` has two warts that
re-edit usage would surface:

- **No pre-population.** Form input values are read from the DOM at
  submit time (lines 590–594) but never WRITTEN from existing
  profile data at open time. A re-edit would start empty.
- **Listener accumulation.** Each `openProfileModal()` call adds a
  fresh `submit` listener (with `{ once: true }`) plus fresh
  `change` listeners on the class and gender selects. First-launch
  only ever opens once so this never bites. Re-edit usage will open
  the modal repeatedly across a session — old listeners may be
  garbage-collected with their elements but the cleanup story is
  fragile. Refactor while we're here.

The refactor splits the function in two:

- `setupProfileModal()` — called **once** during `init()`. Attaches
  all listeners (`submit` on the form, `change` on the class and
  gender selects) to elements that persist across the modal's
  open/close cycles. Listeners stay alive across modal show/hide.
- `openProfileModal()` — called on demand (first-launch and now
  Edit Profile button). Reads the current profile, populates fields,
  shows the modal.

The new pre-population logic in `openProfileModal()`:

```js
const data = loadData();
const profile = (data && data.profile) || {};
document.getElementById('profile-name').value = profile.name || '';
document.getElementById('profile-gender').value = profile.gender || '';
document.getElementById('profile-weight-schedule').value = profile.weightSchedule || '';
document.getElementById('profile-class').value = profile.class || '';
syncProfileTierField();  // re-render tier options for the populated class
document.getElementById('profile-tier').value = profile.tier || '';
```

`syncProfileTierField()` runs after class is set so the tier dropdown
shows the right options for the chosen class; tier value is set
**after** that sync so the value lands in a dropdown that knows about
it.

### Preserve `setupCompletedAt` on re-save

`buildProfileFromFormValues` at `shared.js:83-95` already has the
right fallback — it preserves an incoming `setupCompletedAt` and
stamps a new one only if absent. But the current save handler at
`app.js:589-595` never PASSES `setupCompletedAt` through. Result:
every save stamps a fresh timestamp, overwriting the original.

Fix: add `setupCompletedAt` to the values object passed to the
builder, read from the current profile:

```js
const current = loadData();
const profile = buildProfileFromFormValues({
  name: document.getElementById('profile-name').value,
  gender: document.getElementById('profile-gender').value,
  weightSchedule: document.getElementById('profile-weight-schedule').value,
  class: classSelect.value,
  tier: document.getElementById('profile-tier').value,
  setupCompletedAt: current.profile && current.profile.setupCompletedAt,
});
current.profile = profile;
saveData(current);
```

On first save, `current.profile.setupCompletedAt` is undefined →
builder stamps a fresh one. On re-save, the original timestamp passes
through unchanged.

### Files touched

- `index.html` — add the Edit Profile button at the top of `<main>`.
- `app.js` — split `openProfileModal` into `setupProfileModal` +
  `openProfileModal`, add pre-population logic, fix
  `setupCompletedAt` preservation, wire the Edit Profile button to
  call `openProfileModal`.

### Tests

Three new tests in `tests.js`:

1. `buildProfileFromFormValues: preserves existing setupCompletedAt
   when passed through` — call the builder twice with the same input
   plus an explicit `setupCompletedAt`; assert the value is unchanged
   across both calls.
2. `buildProfileFromFormValues: stamps fresh setupCompletedAt when
   absent` — call with no `setupCompletedAt`; assert the result has a
   recent ISO timestamp.
3. (Integration-style, if the test harness supports DOM) — `openProfileModal:
   pre-populates form fields from current profile`. Skip if the
   harness doesn't render the modal DOM; document as a manual smoke
   test step instead.

Suite count: 376 → 378 (or 379 if test 3 is included).

## Concern 2 — Throws PR card text overlays nudge

### What's wrong today

The plaque text (headline / event / mark / previous) sits a
fingernail too high inside the template's illustrated stone-plaque
region. Same effect on the brass scroll metadata. Oak's eye, viewing
on iPhone Safari at the live URL: "it could pass right now if I
really wanted it to, but I'd like to knock this out while we're doing
these things."

### What changes

Two CSS rules nudged. **Backgrounds and structure stay put** — only
the position of the text-bearing overlays moves:

- `.celebration-card--throw .card-plaque` at `styles.css:1294` —
  `top: 71%` shifts to roughly `top: 73%` (calibrate ±0.5%).
- `.celebration-card--throw .card-scroll` at `styles.css:1365` —
  `top: 90%` shifts to roughly `top: 91%` (calibrate ±0.5%).

The exact percentages are calibration values, not spec values.
ccode iterates against headless screenshots the same way it did for
the original 6c handoff — render a sample throws PR card, eyeball
text-vs-illustration alignment, pick the percentages that land. The
starting points above are the "one fingernail" magnitude Oak called
out; the final value is whatever sits centered in each illustrated
region.

### Scope isolation (carried from 6c)

- The PR shield, silhouette, plaque background, scroll background,
  thistle/laurel framing, mountain backdrop — all from the template
  image — stay put.
- The `.card-headline / -event / -mark / -prev / -wordmark` font
  sizes, weights, colors, line-heights all stay put.
- Lift PR / Goal / Awesome Day cards (using the legacy
  `.celebration-card-*` rules, not `.celebration-card--throw .card-*`)
  are unaffected.

### Files touched

- `styles.css` — two `top:` percentage adjustments.

### Tests

CSS-only positioning change with no behavioral assertion attached.
No automated test added; visual verification is the existing manual
acceptance criterion (smoke-test checklist Step 5 ornate-card
verification still applies after this nudge).

## Concern 3 — Wordmark font shrink

### What's wrong today

The Stone & Standard wordmark at the bottom-center of the ornate
throws PR card (positioned correctly per the prior patch) reads a
touch large against the overall card composition.

### What changes

One CSS rule, one property:

- `.celebration-card--throw .card-wordmark` at `styles.css:1393` —
  `font-size: 0.7rem` becomes `font-size: 0.525rem` (`0.7 × 0.75`).

Everything else on the rule — position, color, letter-spacing,
weight, text-align — stays put.

### Scope isolation

- Legacy `.celebration-card-wordmark` rule (lift PR / Goal /
  Awesome Day cards) untouched.

### Files touched

- `styles.css` — one font-size value.

### Tests

CSS-only; no automated test. Visual verification covered by the
smoke-test checklist.
