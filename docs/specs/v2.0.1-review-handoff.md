## What the spec called for vs. what shipped

### Concern 1 — Edit Profile affordance + modal refactor

**Spec scope:** add a button on the Set PRs & Goals page; split
`openProfileModal` into `setupProfileModal` + `openProfileModal`;
pre-populate fields from current profile; preserve `setupCompletedAt`
on re-save by passing the existing value through to the builder.

**As shipped:** implemented verbatim per spec. No deviations.
 - `setupProfileModal()` attaches all listeners (submit on the form,
   change on the class and gender selects) once during `init()`.
 - `openProfileModal()` reads current data and pre-populates fields in
   the spec-prescribed order (name, gender, weightSchedule, class,
   then `syncProfileTierField()`, then tier value, then
   `syncProfileWeightScheduleField()`).
 - Save handler reads `loadData()` first and passes
   `current.profile && current.profile.setupCompletedAt` to the
   builder, so first save stamps fresh and re-save preserves.
 - `shared.js` is untouched. `buildProfileFromFormValues` already had
   the right fallback; only the call site needed updating.
 - HTML button reads `<span aria-hidden="true">⚙</span> Edit profile`
   — gear is decorative, visible "Edit profile" is the accessible
   name (covers the spec's codex flag about button having an
   accessible name).

**Tests added:** two builder tests in `tests.js`, pinning the
preserve-vs-stamp branches the call site relies on.
 - `buildProfileFromFormValues: preserves existing setupCompletedAt
   across repeated calls` — calls the builder twice with the same
   input + explicit `setupCompletedAt`; asserts the value is unchanged
   across both calls. (Re-edit simulation.)
 - `buildProfileFromFormValues: stamps fresh setupCompletedAt when
   absent` — asserts the stamped timestamp falls within the call
   window (recent, not a default).
 - The spec's optional test #3 (`openProfileModal: pre-populates form
   fields from current profile`) is **skipped per spec** ("Skip if the
   harness doesn't render the modal DOM; document as a manual smoke
   test step instead"). `tests.html` does not include the profile
   modal markup. Manual verification covered by acceptance criteria
   3–5.

**Suite count delta:** 376 → 378.

### Concern 2 — Throws PR card text overlays nudge

**Spec scope:** CSS-only; nudge `.celebration-card--throw .card-plaque`
`top` from 71% down ~2% and `.celebration-card--throw .card-scroll`
`top` from 90% down ~1%. Calibrate ±0.5%. Final value pending visual.

**As shipped — within spec scope:**
 - `.card-plaque top: 71% → 74%`. Spec sketched 73%; live calibration
   bounced through 75% before settling at 74%. Within Oak's eyeball
   acceptance.
 - `.card-scroll top: 90% → 91%`. Spec sketched 91%; iterated to 92%
   and back. Lands at 91%.
 - The CSS comment block above the plaque rule was updated to
   reference `~y=75%` instead of the outdated `~y=67%` anchor.

**As shipped — expanded scope (NOT in original spec):**
The live calibration session surfaced two further concerns Oak chose
to address in the same bundle rather than defer to v2.0.2:

 1. **Subhead retirement + headline rename** (`session.js`).
    Removed the `.card-subhead` element ("New PR Achieved") from
    `buildThrowsPrCard`. Promoted the headline text from
    "Personal Record" to "New PR Achieved" so the plaque shows one
    strong headline instead of two competing labels. The
    `.card-subhead` CSS rule in `styles.css` is **left in place** —
    unused now but harmless; explicit deletion was out of scope.
    The inline comment in `session.js` (formerly explaining the
    mixed-case workaround for the test regex) was updated to match.

 2. **Date typography promotion to match the headline + curve**
    (`session.js` and `styles.css`).
    The date `<span>` was replaced with inline SVG containing a
    `<path>` and `<textPath>` so the type can curve along the brass
    scroll's natural arc. Span-to-SVG is the only practical CSS-
    compatible way to do text-on-path; CSS has no native primitive.
      - SVG carries `viewBox="0 0 200 24"` and
        `preserveAspectRatio="xMidYMid meet"`.
      - Path: `M 10 14 Q 100 32 190 14` — quadratic Bezier smile,
        endpoints at `y=14`, control point at `(100, 32)`.
        Actual visual apex sits at roughly `y=23` (quadratic Bezier
        reaches halfway toward control point). Iterated through
        17/19/22/26 control-point Y values before landing at 32.
      - `overflow: visible` on both the SVG element and its scroll
        container so the deep dip can render outside the SVG box
        without clipping.
      - SVG text styling: `font-size: 14` (viewBox units), weight
        800, letter-spacing `0.06em`, `text-transform: uppercase`,
        `fill: #d4b876` (brass), `stroke: #1a1208` width `1`,
        `paint-order: stroke fill` — so the dark outline reads as a
        halo behind the brass fill instead of thinning the glyphs
        (brass-on-brass without the stroke would vanish).
      - `transform: translateY(-0.4rem)` on the SVG element so the
        curve's sag re-centres on the brass scroll's vertical
        midpoint (transform, not margin, so games/location below
        don't shift).
      - Per-card path id (`card-date-curve-${++cardDatePathCounter}`)
        avoids collisions in any future multi-card render flow.
      - Games and location below stay at the original `.card-meta-line`
        styling (cream supporting text).

**Tests touched:** the Stage 4b regex assertion at the test originally
matching `/Personal Record/` was updated to `/New PR Achieved/` to
follow the headline rename. **One assertion changed, no new tests
added; suite count stays at 378.**

### Concern 3 — Wordmark font shrink

**Spec scope:** CSS-only; one property — `.celebration-card--throw
.card-wordmark font-size: 0.7rem → 0.525rem` (`0.7 × 0.75`).

**As shipped — DEVIATION from spec:**

The spec's shrink reasoning was that the bottom-centre wordmark read
too large against the card composition. Oak then directed a separate
rotation to a bottom-left spine position which changed the visual
math — the rotated spine label has more headroom than the centred
text and can carry more weight. Final state:

 - **Final font-size: `0.7875rem`** (not `0.525rem`). The chain was
   `0.7rem` (shipped) → `0.525rem` (Concern 3 shrink) → `0.7875rem`
   (50% larger after the rotation). End delta vs. shipped v2.0.0:
   `+12.5%`, not `–25%`.
 - **Position: bottom-left, rotated 90° counterclockwise** (book-
   spine pattern, reads bottom-to-top). `left: 4%`, `bottom: 4%`,
   `transform-origin: left bottom`, `transform: rotate(-90deg)`.
 - `text-align: left` and `white-space: nowrap` to keep the rotated
   text as a single clean vertical strip.
 - The wordmark comment was rewritten to describe the rotation and
   pivot-origin rationale.

**This is the single biggest deviation from spec text.** Codex
should treat the rotation + size bump as intentional, owner-
directed, and accepted live by Oak — not as drift from the spec.

## What to ask codex to verify

The spec's codex review prompt at the bottom of `docs/specs/v2.0.1-spec.md`
is still the canonical brief. Hand codex the spec and this handoff
together. The spec's prompt covers the in-scope checks; below are
the additional points the expanded scope introduces:

1. **Subhead removal is intentional.** Codex may flag the orphaned
   `.card-subhead` CSS rule in `styles.css` (no DOM consumer anymore).
   It is deliberately left in place — out-of-scope cleanup. Flag at
   Nit severity at most.

2. **Headline text rename + test regex.** Confirm the regex update in
   `tests.js` (Stage 4b PR card test) matches the new headline text
   `/New PR Achieved/` and that no OTHER assertions in the file still
   look for `/Personal Record/` on the throws PR card. Confirm the
   lift PR card's "New Personal Record" headline text at
   `session.js:798` is unrelated and unchanged.

3. **SVG textPath markup correctness.** Codex should verify:
    - `xmlns` is implicit via `createElementNS`; the SVG namespace
      constant is correct (`http://www.w3.org/2000/svg`).
    - Path id is unique per card render via `++cardDatePathCounter`.
    - `<textPath>` uses `href` (modern) — no `xlink:href` fallback
      provided; this is correct for all current evergreen browsers
      including iOS Safari ≥ 13. Confirm acceptable for project's
      browser baseline (which is "current Safari/Chrome/Firefox").
    - `text-anchor="middle"` + `startOffset="50%"` together centre
      the rendered date at the curve's midpoint.

4. **Wordmark rotation pivot math.** Confirm
   `transform-origin: left bottom` + `transform: rotate(-90deg)` does
   what the comment claims — rotated text rises UP from
   `(left, bottom)` rather than swinging around centre. Confirm
   `left: 4%` is far enough from the card's left edge that the
   rotated text's bounding box (which extends LEFT of the pivot by
   the unrotated element's height ≈ 12px at `0.7875rem` font) does
   not clip the card's left edge across supported card widths.

5. **Date SVG translateY rationale.** The
   `transform: translateY(-0.4rem)` on `.card-meta-date` is intended
   to re-centre the date on the brass scroll illustration. Confirm
   this does not affect flex flow (it shouldn't — `transform` is
   compositing-only) and that games/location below the date stay at
   their flex-column positions.

6. **No regression in non-throws PR cards.** Confirm `buildThrowsPrCard`
   in `session.js` is the only celebration-card builder modified.
   The lift PR / Goal / Awesome Day branches in
   `buildCelebrationCard` and their helper functions should be
   identical to the v2.0 ship.

7. **No data-layer touch.** Spec called the patch "Normal" risk —
   UI + CSS only. Confirm `shared.js` is byte-identical to pre-patch
   and that no `data` or `loadData`/`saveData` shape is altered.
